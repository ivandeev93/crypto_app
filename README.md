Design Decisions
1. Архитектура проекта

Проект построен с использованием принципов Clean Architecture:

domain — бизнес-логика и абстракции (entities, repositories, services)

infrastructure — работа с внешними системами (БД, HTTP-клиенты, ORM)

api — слой доставки (FastAPI роутеры, схемы)

services/tasks — фоновые задачи (Celery)

Такое разделение:

упрощает тестирование,

снижает связанность между слоями,

позволяет менять инфраструктуру без изменения бизнес-логики.

2. Работа с криптобиржей Deribit

Для получения цен используется публичный REST API Deribit.

Решения:

HTTP-клиент реализован отдельно (DeribitClient)

Используется aiohttp для асинхронных запросов

Один ClientSession переиспользуется на все запросы в рамках одной задачи

Это уменьшает накладные расходы на установку соединений и повышает производительность.

3. Асинхронность и Celery

Celery используется для периодического получения цен.

Ключевые решения:

Celery по умолчанию работает синхронно

Асинхронный код выполняется через asyncio.run(...) внутри Celery-таски

Асинхронность используется только там, где она оправдана (HTTP + БД)

Периодический запуск реализован через Celery Beat, встроенный в worker (-B).

Это упрощает инфраструктуру и достаточно для тестового задания.

4. Работа с базой данных

В качестве СУБД используется PostgreSQL.

Решения:

Используется SQLAlchemy Async (AsyncSession)

Работа с БД инкапсулирована в репозиториях

В API используется dependency injection для передачи сессии

Это позволяет:

избежать глобальных переменных,

упростить тестирование,

централизовать работу с транзакциями.

5. Модель данных

Цена валюты хранится в виде:

тикера (ticker)

значения цены (price)

времени получения (timestamp, UNIX)

Используется простая модель без агрегаций, так как задача ориентирована на:

хранение истории цен,

выборку по дате,

получение последнего значения.

6. API

Внешнее API реализовано с помощью FastAPI.

Решения:

Все методы используют HTTP GET

ticker передаётся как обязательный query-параметр

Используются Pydantic-схемы для валидации и сериализации

API покрывает все требования ТЗ:

получение всех данных по валюте

получение последней цены

получение цен за период времени

7. Конфигурация приложения

Настройки вынесены в .env и загружаются через pydantic-settings.

Решения:

Критические зависимости (БД, Redis) обязательны

Внешние сервисы имеют дефолтные значения

Используется кеширование настроек (lru_cache)

Это делает конфигурацию явной и безопасной.

8. Контейнеризация

Приложение разворачивается с помощью Docker Compose.

Решения:

Отдельные контейнеры для:

API

Celery worker

PostgreSQL

Redis

Используется единый образ для API и Celery

Это упрощает деплой и повторяемость окружения.

9. Что можно улучшить в production

В рамках тестового задания намеренно не реализовано:

отдельный сервис Celery Beat

retry-политики для внешних запросов

алерты и мониторинг

миграции БД (Alembic)

кэширование результатов

Эти улучшения логично добавлять на следующем этапе развития проекта.
